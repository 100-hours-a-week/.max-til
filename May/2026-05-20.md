## 🚀 Today I Learned

## | OOP

- OOP(Object Oriented Programming)는 객체지향 프로그래밍을 의미한다.

- 프로그램을 단순히 기능 중심으로 작성하는 것이 아니라, 데이터와 기능을 가진 객체 단위로 나누어 설계하는 방식이다.

- 객체는 자신의 상태를 나타내는 필드와 동작을 나타내는 메서드를 가질 수 있다.

### 왜 쓰는가?

- 프로그램의 규모가 커질수록 모든 코드를 한 곳에 작성하면 관리하기 어려워진다.

- 객체지향을 사용하면 역할별로 코드를 나눌 수 있기 때문에 코드의 구조를 이해하기 쉬워진다.

- 또한 기능을 수정하거나 확장할 때 관련된 객체 중심으로 변경할 수 있어 유지보수에 유리하다.

<br>

## | SOLID

- SOLID는 객체지향 설계에서 유지보수하기 쉽고 확장 가능한 코드를 만들기 위한 5가지 설계 원칙이다.

- 단순히 문법을 잘 쓰는 것이 아니라, 변경에 강한 구조를 만들기 위한 설계 기준이라고 볼 수 있다.

### 왜 쓰는가?

- 객체지향으로 코드를 작성한다고 해서 무조건 좋은 구조가 되는 것은 아니다.

- 책임이 섞이거나, 기능 하나를 추가할 때 기존 코드를 계속 수정해야 한다면 유지보수가 어려워진다.

- SOLID 원칙을 지키면 코드의 책임을 명확히 나누고, 기능 변경이나 확장에 더 유연하게 대응할 수 있다.

<br>

## | SRP(Single Responsibility Principle)

- SRP는 단일 책임 원칙이다.

- 하나의 클래스는 하나의 책임만 가져야 하며, 클래스가 수정되는 이유도 하나여야 한다는 원칙이다.

- 즉, 내가 만든 클래스가 “몇 가지 이유로 수정될 수 있는가?”라고 생각했을 때, 한 가지 이유로만 수정되어야 한다.

### 왜 쓰는가?

- 하나의 클래스가 너무 많은 역할을 가지면, 작은 기능을 수정해도 다른 기능에 영향을 줄 수 있다.

- 책임을 분리하면 코드의 역할이 명확해지고, 수정해야 하는 범위를 줄일 수 있다.


<br>

## | OCP

- OCP는 개방-폐쇄 원칙이다.

- 소프트웨어는 기능 확장에는 열려 있어야 하고, 기존 코드 변경에는 닫혀 있어야 한다는 원칙이다.

- 새로운 기능을 추가할 때 기존 코드를 직접 수정하기보다, 새로운 클래스를 추가하거나 확장하는 방식으로 설계해야 한다.

### 왜 쓰는가?

- 기능을 추가할 때마다 기존 코드를 계속 수정하면 오류가 발생할 가능성이 높아진다.

- 기존 코드는 최대한 안정적으로 유지하고, 새로운 기능은 확장으로 추가하는 구조가 더 안전하다.

- 또한 코드를 변경할 때 다른 곳에서도 영향을 줄 수 있기 때문에 이러한 원칙을 사용하는 것이다.


<br>

## | JVM

- JVM(Java Virtual Machine)은 자바 바이트코드를 실행해주는 가상의 실행 환경이다.

- 자바 코드는 바로 운영체제에서 실행되는 것이 아니라, 먼저 바이트코드로 변환된 뒤 JVM 위에서 실행된다.

- JVM은 운영체제에 맞게 바이트코드를 실행해주기 때문에 자바는 한 번 작성한 코드를 여러 환경에서 실행할 수 있다.

### 왜 쓰는가?

- 운영체제마다 실행 방식이 다르기 때문에, 코드가 운영체제에 직접 의존하면 환경별로 따로 작성해야 할 수 있다.

- JVM은 자바 코드와 운영체제 사이에서 중간 역할을 하며, 바이트코드를 각 운영체제에 맞게 실행해준다.

- 한번 코드를 작성하면 어느곳에서든 실행이 가능하기 때문이다.


<br>

## | 클래스 로더

- 클래스 로더(Class Loader)는 `.class` 파일을 JVM 메모리로 불러오는 역할을 한다.

- 자바 소스 파일은 컴파일 후 바이트코드가 담긴 `.class` 파일이 된다.

- 클래스 로더는 이 `.class` 파일을 읽어서 JVM의 Runtime Data Area에 올린다.

### 왜 쓰는가?

- `.class` 파일이 디스크에만 있으면 프로그램을 실행할 수 없다.

- 실행하려면 클래스 정보가 JVM 메모리에 올라가야 한다.

- 클래스 로더는 필요한 클래스를 메모리에 적재해서 실행 준비를 해주는 역할을 한다.

<br>

---

## | 실행 엔진

- 실행 엔진(Execution Engine)은 Runtime Data Area에 올라온 바이트코드를 실제로 실행하는 역할을 한다.

- JVM은 바이트코드를 읽은 뒤 실행 엔진을 통해 프로그램을 동작시킨다.

- 실행 엔진에는 인터프리터, JIT 컴파일러, 가비지 컬렉터 등이 포함된다.

- 실행 엔진은 바이트코드를 해석하거나 기계어로 바꾸면서 프로그램을 실행한다.

### 왜 쓰는가?

- 바이트코드는 컴퓨터가 바로 실행할 수 있는 완전한 기계어가 아니다.

- 따라서 JVM 내부에서 바이트코드를 실제 실행 가능한 형태로 처리하는 과정이 필요하기 때문이다.

<br>

---

## | JIT

- JIT(Just-In-Time Compiler)는 자주 실행되는 바이트코드를 기계어로 변환해 실행 속도를 높이는 컴파일러이다.

- 인터프리터는 바이트코드를 한 줄씩 읽어서 실행하지만, 같은 코드가 반복되면 비효율적일 수 있다.

- JIT는 자주 사용되는 코드를 미리 기계어로 바꿔두어 이후 실행 속도를 높인다.

### 왜 쓰는가?

- 바이트코드를 매번 한 줄씩 해석하면 실행 속도가 느려질 수 있다.

- 반복적으로 실행되는 코드는 기계어로 변환해두면 더 빠르게 실행할 수 있다.

- 즉, JIT는 자바 프로그램의 성능을 높이기 위해 사용된다.

<br>

---

## | Runtime Data Area

- Runtime Data Area는 자바 프로그램이 실행되는 동안 필요한 데이터가 저장되는 JVM의 메모리 공간이다.

- 클래스 정보, 객체, 메서드 실행 정보, 현재 실행 위치 등이 이 영역에 저장된다.

- 대표적으로 Method Area, Heap, PC Register, JVM Stack, Native Method Stack이 있다.

### 왜 쓰는가?

- 프로그램을 실행하려면 필요한 코드와 데이터가 메모리에 올라가 있어야 한다.

- `.class` 파일이 만들어졌다고 바로 실행되는 것이 아니라, JVM이 실행에 필요한 정보를 메모리에 저장해야 한다.

- Runtime Data Area는 JVM이 프로그램을 실행하기 위해 사용하는 작업 공간이라고 볼 수 있다.

<br>

---

## | Method Area

- Method Area는 클래스 정보, 메서드 정보, static 변수, 상수 정보 등이 저장되는 영역이다.

- 클래스 로더가 `.class` 파일을 읽어오면 클래스와 관련된 정보가 Method Area에 저장된다.

- 쉽게 말해 클래스의 설계도와 같은 정보가 저장되는 공간이다.

### 왜 쓰는가?

- JVM이 객체를 만들거나 메서드를 실행하려면 클래스에 대한 정보가 필요하다.

- 어떤 필드가 있는지, 어떤 메서드가 있는지, static 변수는 무엇인지 알아야 프로그램을 실행할 수 있다.


<br>

---

## | Native Method Stack

- Native Method Stack은 자바가 아닌 다른 언어로 작성된 네이티브 메서드의 실행 정보를 저장하는 영역이다.

- 자바는 필요할 때 C나 C++ 같은 언어로 작성된 기능을 호출할 수 있다.

- 이런 네이티브 메서드가 실행될 때 필요한 정보가 Native Method Stack에 저장된다.

### 왜 쓰는가?

- 자바 프로그램이 운영체제 기능이나 외부 라이브러리를 사용할 때 자바가 아닌 코드와 연결될 수 있다.

- 이때 네이티브 코드의 실행 정보를 따로 관리할 공간이 필요하다.

- Native Method Stack은 자바 외부 코드 실행을 지원하기 위한 메모리 영역이다.

<br>

## 💭 오늘의 회고


오늘은 객체지향 프로그래밍의 기본 개념인 OOP와 객체지향 설계 원칙인 SOLID, 그리고 자바 프로그램이 실행되는 구조인 JVM에 대해 정리해보았다. 처음에는 객체지향이 단순히 클래스를 만들고 객체를 생성하는 것이라고 생각했지만, 실제로는 역할과 책임을 나누어 유지보수하기 좋은 구조를 만드는 방식이라는 것을 알 수 있었다.

특히 SOLID 원칙을 공부하면서 좋은 코드는 단순히 동작만 하는 코드가 아니라, 변경과 확장에 강한 코드라는 점을 이해할 수 있었다. SRP는 하나의 클래스가 하나의 책임을 가지도록 하고, OCP는 기존 코드를 최대한 수정하지 않고 기능을 확장할 수 있도록 도와준다. 또한 DIP처럼 구현체가 아니라 인터페이스에 의존하는 구조는 Spring에서 의존성 주입을 사용하는 이유와도 연결된다고 느꼈다. 나는 학교에서 배울때는 이 원칙을 무조건 지켜야하는데 실제로는 코드를 어떻게 자야 이 원칙을 지킬 수 있을까 라는 고민을 많이 했었다. 비록 이번 공부를 통해서도 실제 코드에 적용하기 위해서는 많은 고민을 해야할테지만 그래도 어느정도의 기초를 닦았다고 생각한다.

JVM을 공부하면서는 자바 코드가 바로 실행되는 것이 아니라, `.java` 파일이 컴파일되어 `.class` 바이트코드가 되고, 클래스 로더가 이를 JVM 메모리에 올린 뒤 실행 엔진이 실행한다는 흐름을 이해할 수 있었다. Runtime Data Area에는 클래스 정보, 객체, 메서드 실행 정보 등이 저장되며, 이 공간이 있어야 JVM이 프로그램을 실행할 수 있다는 점도 알게 되었다. 지금까지는 JVM이 어떤 애인지만 배웠지 이 안에서 어떤 방식으로 흘러가는지는 배우지도 않았고 관심도 없었던 것 같다. 강의를 들으면서 어떤 기술의 실행과정을 배운다는것이 오히려 이것을 이해하는데 큰 도움이 된다는 것을 깨달았다.
