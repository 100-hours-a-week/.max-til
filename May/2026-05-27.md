## 🚀 Today I Learned

## | HTTP 자격 증명 헤더

- HTTP 자격 증명 헤더는 클라이언트가 서버에게 자신의 인증 정보를 전달할 때 사용하는 HTTP 헤더이다.

- 대표적으로 Authorization 헤더가 있으며, 클라이언트는 이 헤더에 인증 방식과 인증 값을 함께 담아 서버로 보낸다.

### 왜 쓰는가?

- 서버는 요청을 보낸 사용자가 누구인지 확인해야 하기 때문이다.

- HTTP는 기본적으로 Stateless하기 때문에 서버는 이전 요청에서 로그인했는지 자동으로 기억하지 않는다.

- 그래서 클라이언트는 API를 요청할 때마다 자신이 인증된 사용자라는 정보를 헤더에 담아 보내야 한다.

## | Basic Authentication

- Basic Authentication은 HTTP에서 제공하는 가장 기본적인 인증 방식이다.

- 사용자의 아이디와 비밀번호를 아이디:비밀번호 형태로 만든 뒤 Base64로 인코딩해서 Authorization 헤더에 담아 보낸다.

### 왜 쓰는가?

- 인증 구조가 단순해서 빠르게 구현할 수 있기 때문이다.

- 별도의 로그인 세션이나 토큰 발급 구조 없이도 아이디와 비밀번호만으로 서버가 사용자를 확인할 수 있다.

- 그래서 실제 서비스에서는 반드시 HTTPS와 함께 사용해야 하며, 일반적인 웹 서비스 인증 방식으로는 잘 사용하지 않는다.


<br>

## | Bearer Token

- Bearer Token은 토큰을 가진 사람이 권한을 가진 사용자라고 판단하는 인증 방식이다.

- 클라이언트는 서버로부터 발급받은 토큰을 가지고 있다가 API 요청 시 Authorization 헤더에 담아 보낸다.


### 왜 쓰는가?

- 매 요청마다 아이디와 비밀번호를 보내지 않아도 되기 때문이다.

- 사용자는 한 번 로그인해서 토큰을 발급받고, 이후에는 토큰만 보내면 인증된 요청을 할 수 있다.

- 또한 서버 입장에서는 토큰을 기준으로 사용자를 식별하고 권한을 확인할 수 있다.


<br>

## | JWT

- JWT는 JSON Web Token의 약자로, 사용자 정보와 권한 정보를 JSON 형태로 담을 수 있는 토큰 형식이다.

- JWT는 보통 Header.Payload.Signature 구조로 이루어져 있다.

- Header에는 토큰 타입과 서명 알고리즘 정보가 들어간다.

- Payload에는 사용자 id, 권한, 만료 시간 같은 정보가 들어갈 수 있다.

- Signature는 토큰이 서버에서 발급된 이후 위조되거나 변조되지 않았는지 확인하기 위한 서명 값이다.

### 왜 쓰는가?

- 모든 통신에서 안정성보다 속도가 더 중요한 경우가 있다.

- 예를 들어 실시간 영상 스트리밍이나 게임처럼 약간의 데이터 손실보다 빠른 전송이 중요한 상황에서는 연결 과정이 오히려 부담이 될 수 있다.

- 그래서 Connectless 방식은 빠른 전송이 중요한 통신에서 사용된다.



<br>

## | Access Token

- Access Token은 사용자가 API에 접근할 때 인증을 증명하기 위해 사용하는 토큰이다.

- 클라이언트는 로그인 성공 후 Access Token을 발급받고, 이후 API 요청마다 이 토큰을 함께 보낸다.

- 서버는 Access Token을 검증해서 사용자가 누구인지, 해당 API에 접근할 권한이 있는지 확인한다.

### 왜 쓰는가?

- 사용자가 매 요청마다 아이디와 비밀번호를 보내지 않고도 인증된 요청을 할 수 있게 하기 때문이다.

- 서버는 Access Token을 통해 사용자를 식별하고, 해당 사용자가 요청한 작업을 수행할 권한이 있는지 확인할 수 있다.

<br>

## | Refresh Token

- Refresh Token은 Access Token이 만료되었을 때 새로운 Access Token을 발급받기 위해 사용하는 토큰이다.

- Access Token은 API 요청에 자주 사용되지만, Refresh Token은 Access Token을 재발급받을 때 주로 사용된다.

- 일반적으로 Refresh Token은 Access Token보다 만료 시간이 길다.

### 왜 쓰는가?

- Access Token의 수명을 짧게 가져가면서도 사용자가 자주 다시 로그인하지 않게 하기 때문이다.

- Access Token을 짧게 유지하면 탈취되었을 때 피해 시간을 줄일 수 있다.

- 하지만 너무 자주 만료되면 사용자는 계속 로그인해야 해서 불편하다.

- Refresh Token을 사용하면 보안성과 사용자 편의성을 함께 가져갈 수 있기 때문이다.


<br>

## | RTR

- RTR은 Refresh Token Rotation의 약자이다.

- Refresh Token을 한 번 사용할 때마다 기존 Refresh Token을 폐기하고 새로운 Refresh Token을 발급하는 방식이다.


### 왜 쓰는가?

- Refresh Token이 탈취되었을 때 피해를 줄이기 위해서이다.

- 만약 Refresh Token을 계속 재사용할 수 있다면, 공격자가 한 번 탈취한 Refresh Token으로 계속 Access Token을 발급받을 수 있다.

- 쿠키를 사용하면 브라우저가 요청마다 특정 정보를 함께 보내기 때문에, 서버가 사용자를 구분하거나 상태를 이어서 처리할 수 있다.

## 💭 오늘의 회고

오늘은 HTTP 인증 구조에서 자주 등장하는 개념들을 정리해보았다. 처음에는 토큰, JWT, Access Token, Refresh Token이 비슷하게 느껴졌지만, 각각의 역할이 다르다는 점을 이해할 수 있었다.

특히 Bearer Token은 특정 토큰 형식이 아니라 Authorization 헤더에 토큰을 담아 보내는 인증 방식이고, JWT는 그 토큰을 표현하는 하나의 형식이라는 점이 중요했다.

인증 구조는 단순히 로그인 기능을 만드는 문제가 아니라, 토큰이 탈취되었을 때 어떤 위험이 생기고 어떻게 줄일 수 있는지까지 고려해야 하는 영역이라는 생각이 들었다

앞으로 JWT 기반인증을 구현할때는 Access Token과 Refresh Token을 분리하여 보안적으로도 신경을 써서 개발을 해볼 생각이다.
