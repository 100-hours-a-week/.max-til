## 🚀 Today I Learned

## | 서블릿

- 서블릿(Servlet)은 자바에서 웹 요청과 응답을 처리하기 위한 기술이다.

- 클라이언트가 서버에 HTTP 요청을 보내면, 서블릿은 그 요청을 받아서 처리하고 다시 HTTP 응답을 만들어 돌려준다.

- Spring MVC도 내부적으로는 서블릿 기반으로 동작하며, 그 중심에는 DispatcherServlet이 있다.

### 왜 쓰는가?

- 웹은 사용자의 요청을 받고, 그 요청에 맞는 결과를 응답해야한다. 이때 HTTP 요청과 응답을 관리하고 다루는 구조가 필요하기 때문이다.

- 개발자가 모든것을 직접 구현하지 않아도 자동으로 요청,응답을 처리해주는 기능을 사용할 수 있게 해준다.

## | Spring Web MVC 패턴

- Spring Web MVC는 웹 애플리케이션을 Model, View, Controller로 나누어 개발하는 구조이다.

- MVC는 각각의 역할을 분리해서 웹 애플리케이션을 더 이해하기 쉽고 유지보수하기 쉽게 만든다.

- Controller는 필요한 데이터를 처리하고, 그 결과를 Model에 담거나 응답 데이터로 반환한다.

- View는 예전에는 HTML로 많이 응답데이터로 주었지만 지금은 프론트에서 기능을 처리하면서 데이터만 주는 경우도 많이 있다.


### 왜 쓰는가?

- 웹 애플리케이션의 모든 코드를 한 곳에 작성하면 화면 처리, 데이터 처리, 요청 처리 로직이 섞여서 관리하기 어려워진다.

- MVC 패턴을 사용하면 요청을 처리하는 부분, 데이터를 담는 부분, 화면을 보여주는 부분을 분리할 수 있다.

- 역할이 분리되면 코드의 흐름을 이해하기 쉽고, 특정 기능을 수정할 때 수정 범위를 줄일 수 있다.

<br>

## | Model

- Model은 화면이나 응답에 필요한 데이터를 처리하고 View에 전달하기 위한 데이터를 담는다.

### 왜 쓰는가?

- Controller는 데이터를 준비하고, View는 그 데이터를 보여주는 역할에 집중할 수 있다.

<br>

---

## | View

- 사용자에게 보여지는 화면을 담당한다.

- Controller가 Model에 데이터를 담아 전달하면, View는 그 데이터를 사용해 사용자에게 보여줄 화면을 만든다.

### 왜 쓰는가?

- 사용자는 서버 내부 로직을 직접 보는 것이 아니라, 처리 결과가 반영된 화면이나 응답을 보게 된다

- 화면 출력 역할을 View로 분리하면 Controller가 화면 코드까지 책임지지 않아도 되기 때문이다.


<br>

---

## | Controller

- Controller는 클라이언트의 요청을 받아서 처리 흐름을 제어하는 역할을 한다.

- Controller는 요청 URL과 HTTP Method에 맞는 메서드를 실행하고, Service를 호출해 필요한 작업을 처리한다.

### 왜 쓰는가?

- 클라이언트 요청이 들어왔을 때 어떤 기능을 실행해야 하는지 결정하는 부분이 필요하다.

- 또한 요청 처리 흐름을 Controller에 모아두면 API나 화면 요청의 진입점을 쉽게 파악할 수 있다.

- 결국 Controller는 웹 요청을 애플리케이션 내부 로직과 연결하는 입구 역할을 한다.

<br>

---

## | Handler Mapping

- Handler Mapping은 클라이언트 요청을 처리할 Controller를 찾아주는 역할을 한다.

- 여기서 Handler는 요청을 실제로 처리할 수 있는 대상, 보통 Controller의 메서드를 의미한다.

### 왜 쓰는가?

- 웹 애플리케이션에는 수많은 URL과 Controller 메서드가 존재할 수 있다. 

- 따라서 요청이 들어올때마다 어떤 Controller가 처리해야되는지 찾는 과정이 필요하기 때문이다.

<br>

---

## | DispatcherServlet

- DispatcherServlet은 Spring MVC에서 모든 웹 요청을 가장 먼저 받는 핵심 서블릿이다

- 클라이언트 요청이 들어오면 DispatcherServlet이 요청을 받고, Handler Mapping을 통해 알맞은 Controller를 찾는다.

- 이후 Controller를 실행하고, 처리 결과에 따라 View를 선택하거나 JSON 응답을 반환한다.

### 왜 쓰는가?

- 요청마다 각각의 서블릿을 따로 만들고 관리하면 구조가 복잡해진다.

- DispatcherServlet이 모든 요청의 입구 역할을 하면 요청 처리 흐름을 한 곳에서 관리할 수 있다.

- 공통 처리, Controller 탐색, View 연결, 예외 처리 같은 흐름을 일관되게 구성할 수 있다.
