## 🚀 Today I Learned

## | NoSQL

- NoSQL은 `Not Only SQL`의 약자로, SQL만 사용하는 관계형 데이터베이스가 아니라 다양한 데이터 저장 방식을 포함하는 데이터베이스를 의미한다.

- NoSQL은 데이터를 반드시 테이블, 행, 열 구조로만 저장하지 않는다.

- 데이터의 형태에 따라 `Key-Value`, `Document`, `Column-Oriented`, `Graph` 같은 여러 방식으로 저장할 수 있다.

### 왜 쓰는가?

- 서비스에서 다루는 데이터가 항상 정해진 테이블 구조로만 표현되지는 않기 때문이다.

- 예를 들어 게시글, 댓글, 로그, 채팅 메시지, 사용자 행동 데이터처럼 구조가 자주 바뀌거나 대량으로 쌓이는 데이터는 RDB만으로 처리하기 부담스러울 수 있다.

- 그래서 데이터 구조를 더 유연하게 가져가고, 대규모 데이터를 빠르게 읽고 쓰며, 서버를 여러 대로 늘려 수평 확장하기 위해 NoSQL을 사용한다.

## | 중앙 Stream

- 중앙 Stream은 데이터를 여러 곳에서 각각 따로 처리하지 않고, 하나의 중심 데이터 흐름에 모아서 처리하는 구조를 의미한다.

- 여러 서비스나 서버에서 발생하는 이벤트, 로그, 메시지 등을 하나의 Stream으로 모으면 이후에 필요한 시스템들이 같은 흐름을 기준으로 데이터를 소비할 수 있다.

- 예를 들어 주문 발생, 결제 완료, 알림 발송 같은 이벤트를 중앙 Stream에 모아두면 각 서비스가 필요한 이벤트만 가져가 처리할 수 있다.

### 왜 쓰는가?

- 서비스가 커질수록 데이터가 여러 곳에서 동시에 발생하고, 이를 각각 따로 연결하면 구조가 복잡해지기 때문이다.

- 중앙 Stream을 사용하면 데이터 흐름을 한 곳에 모아 관리할 수 있어 시스템 간 연결이 단순해진다.

- 또한 이벤트를 기준으로 여러 시스템이 독립적으로 동작할 수 있기 때문에 확장성과 유지보수성이 좋아진다.

## | NoSQL의 특징

- NoSQL은 관계형 테이블 구조에 얽매이지 않고 데이터를 더 유연한 방식으로 저장한다.

- 대표적인 저장 방식으로는 `Key-Value`, `Document`, `Column-Oriented`, `Graph`가 있다.

- JOIN이나 복잡한 트랜잭션보다는 단순 조회, 빠른 읽기/쓰기, 수평 확장에 초점을 둔다.

- 데이터 중복을 어느 정도 허용하더라도 조회 성능과 확장성을 우선하는 경우가 많다.

### 왜 쓰는가?

- 모든 데이터가 관계 중심으로 설계될 필요는 없기 때문이다.

- 예를 들어 캐시, 세션, 로그, 추천 데이터, 채팅 메시지처럼 빠르게 저장하고 빠르게 조회하는 것이 중요한 데이터가 있다.

- 이런 경우에는 엄격한 관계와 정규화보다 속도, 확장성, 유연성이 더 중요하기 때문에 NoSQL을 사용한다.

## | Key-Value Store

- Key-Value Store는 데이터를 `key`와 `value` 한 쌍으로 저장하는 NoSQL 방식이다.

- key를 통해 value를 바로 찾는 구조이기 때문에 매우 단순하고 빠르다.

- 대표적으로 Redis, Aerospike, DynamoDB 등이 있다.

### 왜 쓰는가?

- 특정 key로 데이터를 빠르게 조회해야 하는 상황이 많기 때문이다.

- 예를 들어 로그인 세션, 인증 토큰, 인기 게시글 캐시처럼 자주 조회되는 데이터를 매번 DB에서 가져오면 성능이 떨어진다.

- 이런 경우 Key-Value Store를 사용하면 빠른 읽기/쓰기 성능을 얻을 수 있고, DB 부하도 줄일 수 있기 때문에 사용한다.

## | In-Memory

- In-Memory는 데이터를 디스크가 아니라 메모리에 저장하거나 메모리 중심으로 처리하는 방식이다.

- 디스크보다 메모리 접근 속도가 훨씬 빠르기 때문에 매우 빠른 응답 속도를 낼 수 있다.

- Redis 같은 저장소가 대표적인 In-Memory 기반 데이터 저장소이다.

- 다만 메모리는 휘발성이 있기 때문에 서버가 종료되거나 장애가 발생하면 데이터가 사라질 수 있다.

- 그래서 사라져도 복구 가능하거나, 짧은 시간 동안만 필요한 데이터에 적합하다.

### 왜 쓰는가?

- 자주 접근하는 데이터를 매번 디스크나 DB에서 읽어오면 응답 속도가 느려지기 때문이다.

- 예를 들어 캐시, 세션, 임시 토큰, 실시간 랭킹처럼 빠르게 읽고 쓰는 데이터는 메모리에 두는 것이 유리하다.

- 이런 데이터는 속도가 매우 중요하고 데이터 크기가 비교적 작기 때문에 In-Memory 방식을 사용한다.

## | Document Store

- Document Store는 데이터를 JSON, BSON, XML 같은 문서 형태로 저장하는 NoSQL 방식이다.

- 하나의 문서 안에 관련된 데이터를 함께 저장할 수 있다.

- 대표적으로 MongoDB, CouchDB 등이 있다.

- RDB처럼 정해진 테이블 컬럼에 맞춰 데이터를 넣는 방식보다 스키마 변경이 유연하다.

- 게시글, 상품 카탈로그, 블로그, 채팅 메시지처럼 데이터 구조가 자주 바뀌거나 객체 단위로 다루기 좋은 데이터에 적합하다.

### 왜 쓰는가?

- 서비스 데이터 구조가 항상 고정되어 있지는 않기 때문이다.

- 예를 들어 상품마다 속성이 다르거나, 게시글마다 첨부 데이터 구조가 달라질 수 있다.

- 이런 경우 RDB에서는 컬럼을 계속 추가하거나 테이블 구조를 바꿔야 하지만, Document Store는 문서 단위로 유연하게 저장할 수 있기 때문에 사용한다.



<br>

## | Column-Oriented Store

- Column-Oriented Store는 데이터를 컬럼 중심으로 저장하거나 컬럼 그룹 단위로 관리하는 NoSQL 방식이다.

- 대용량 데이터 저장과 빠른 조회에 적합하다.

- 대표적으로 Cassandra, HBase 등이 있다.

- 로그, 시계열 데이터, 분석 데이터처럼 대량으로 쌓이고 특정 조건으로 빠르게 조회해야 하는 데이터에 많이 사용된다.

- RDB처럼 자유로운 JOIN이나 복잡한 쿼리보다는, 미리 정해진 조회 패턴에 맞춰 대량 데이터를 빠르게 처리하는 데 초점을 둔다.

### 왜 쓰는가?

- 대규모 데이터를 빠르게 저장하고, 특정 패턴으로 효율적으로 조회해야 하기 때문이다.

- 예를 들어 사용자 행동 로그, 센서 데이터, 추천 시스템 데이터는 계속해서 대량으로 쌓인다.

- 이런 데이터를 일반 RDB에 모두 저장하고 조회하면 성능 부담이 커질 수 있기 때문에, 분산 저장과 대량 처리에 강한 Column-Oriented Store를 사용한다.

## | Graph Store

- Graph Store는 데이터를 노드와 엣지 구조로 저장하는 NoSQL 방식이다.

- 노드는 데이터의 대상이고, 엣지는 노드 사이의 관계를 의미한다.

- 예를 들어 사용자, 게시글, 상품은 노드가 될 수 있고, 친구 관계, 좋아요, 팔로우는 엣지가 될 수 있다.

- 대표적으로 Neo4j, Amazon Neptune 등이 있다.

- 소셜 네트워크, 추천 엔진, 지식 그래프처럼 관계 탐색이 중요한 서비스에 적합하다.

### 왜 쓰는가?

- 데이터 자체보다 데이터 사이의 관계를 빠르게 탐색해야 하는 경우가 있기 때문이다.

- 예를 들어 “내 친구의 친구”, “이 사용자가 좋아할 만한 상품”, “A와 B가 어떤 관계로 연결되어 있는가” 같은 조회는 관계 탐색이 핵심이다.

- 이런 경우 RDB에서는 여러 번 JOIN이 필요할 수 있지만, Graph Store는 노드와 엣지를 따라가며 관계를 자연스럽게 탐색할 수 있기 때문에 사용한다.

## | Local Cache

- Local Cache는 각각의 서버 인스턴스 내부에 독립적으로 존재하는 캐시이다.

- 보통 애플리케이션 서버의 메모리에 데이터를 저장한다.

- 서버 내부에 캐시가 있기 때문에 데이터를 가져올 때 별도의 네트워크 통신이 필요 없다.

- 그래서 접근 속도가 매우 빠르고, 구현도 비교적 단순하다.

- 하지만 서버가 여러 대로 늘어나면 각 서버가 서로 다른 캐시 데이터를 가질 수 있다.(일관성이 없을 수 있다.)

- 예를 들어 WAS 1에는 인기 게시글 1위가 A로 저장되어 있고, WAS 2에는 인기 게시글 1위가 B로 저장되어 있을 수 있다.

- 인기 게시글, 임시 계산 결과, 자주 바뀌지 않는 설정값처럼 데이터가 조금 늦게 반영되어도 문제가 크지 않은 경우에 적합하다.

### 왜 쓰는가?

- 데이터 접근 속도를 최대한 빠르게 가져가야 하는 경우가 있기 때문이다.

- 예를 들어 인기 게시글 목록이나 임시 계산 결과를 매번 DB에서 조회하면 불필요한 DB 부하가 생기고 응답 속도도 느려질 수 있다.

- 이런 경우 Local Cache를 사용하면 서버 메모리에서 바로 데이터를 가져올 수 있기 때문에 빠른 응답이 가능하다.

- 다만 서버마다 캐시 값이 달라질 수 있으므로, 로그인 세션, 상품 가격, 재고 수량처럼 모든 서버가 같은 값을 봐야 하는 데이터에는 적합하지 않다.

## | Global Cache

- Global Cache는 여러 서버 인스턴스가 함께 공유하는 캐시이다.

- 보통 Redis, Memcached 같은 별도의 캐시 서버를 두고 사용한다.

- 모든 WAS가 같은 캐시 서버를 바라보기 때문에 캐시 데이터를 공유할 수 있다.

- 서버가 여러 대여도 같은 캐시 데이터를 사용할 수 있기 때문에 Local Cache보다 일관성을 유지하기 쉽다.

- 다만 외부 캐시 서버에 접근해야 하므로 네트워크 통신이 필요하고, Local Cache보다 상대적으로 느릴 수 있다.

- 로그인 세션, 사용자 권한, 상품 가격, 재고 수량, 쿠폰 사용 여부처럼 어느 서버에서 요청을 처리하더라도 같은 값이 필요한 경우에 적합하다.

### 왜 쓰는가?

- 여러 서버가 같은 캐시 데이터를 공유해야 하는 경우가 있기 때문이다.

- 예를 들어 사용자의 요청이 로드밸런서를 통해 WAS 1로 갈 수도 있고, WAS 2로 갈 수도 있다.

- 이때 로그인 세션 정보가 각 서버의 Local Cache에 따로 저장되어 있으면, 어떤 서버에서는 로그인 상태이고 다른 서버에서는 로그아웃 상태처럼 보일 수 있다.

- 그래서 하나의 공통 캐시 저장소를 두고 모든 서버가 같은 데이터를 보도록 하기 위해 Global Cache를 사용한다.

## 💭 오늘의 회고

오늘은 NoSQL의 개념과 종류, 데이터베이스 확장 방식, 그리고 캐시 구조에 대해 정리해보았다. 처음에는 NoSQL을 단순히 “SQL을 사용하지 않는 데이터베이스” 정도로만 생각했지만, 실제로는 데이터의 형태와 서비스 요구사항에 따라 저장 방식을 다르게 선택하는 개념이라는 점을 알 수 있었다.

특히 RDB와 NoSQL의 차이를 비교하면서, 둘 중 하나가 무조건 더 좋은 것이 아니라 사용 목적이 다르다는 점이 중요하게 느껴졌다. NoSQL은 스키마가 유연하고, 대규모 데이터 처리와 수평 확장에 더 잘 맞는 구조를 가진다. 결국 데이터베이스를 선택할 때는 “이 데이터가 관계 중심인가?”, “정확성이 중요한가?”, “빠른 조회와 확장이 중요한가?”를 먼저 고민해야 한다고 느꼈다.

캐시 부분에서는 Local Cache와 Global Cache의 차이가 가장 중요했다. Local Cache는 서버 내부 메모리를 사용하기 때문에 빠르지만, 서버가 여러 대일 때 값이 달라질 수 있다. 반대로 Global Cache는 여러 서버가 같은 캐시를 공유할 수 있어 일관성에는 유리하지만, 네트워크 비용이 발생한다. 그래서 캐시는 무조건 빠르게 만들기 위한 기술이 아니라, 데이터의 일관성과 속도 사이에서 적절한 선택을 해야 하는 기술이라는 점을 알게 되었다. 캐시라는것에 대해서 그냥 값을 빠르게 가져온다 정도만 알았지 로컬인지 글로벌인지에 따라서도 완전 다르게 활용될 수 있구나 라는 것도 알게되었다.

개발자는 어떤 기술을 써야할 지 고민할때 기술의 장점을 보는 것도 좋지만 그 기술의 단점도 함께 보면서 가성비 있는 선택을 해야한다. 앞으로 더 많이 공부하여 여러 방면에서 고민하고 선택하는 개발자가 되고싶다.
