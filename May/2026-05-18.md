## 🚀 Today I Learned

## | Spring

- Spring은 자바 애플리케이션을 더 쉽게 개발할 수 있도록 도와주는 프레임워크이다.

- 객체 생성, 의존성 관리, 트랜잭션 처리, 보안, 웹 개발 등 애플리케이션 개발에 필요한 여러 기능을 제공한다.

- 개발자가 모든 객체를 직접 만들고 연결하는 것이 아니라, Spring이 객체를 관리하고 필요한 곳에 주입해주는 구조를 제공한다.

### 왜 쓰는가?

- 자바로 애플리케이션을 만들다 보면 객체 생성, 객체 간 연결, 설정 관리, 반복되는 공통 로직 처리 등이 복잡해진다.

- Spring을 사용하면 이런 반복적이고 복잡한 부분을 프레임워크가 대신 관리해주기 때문에 개발자는 핵심 비즈니스 로직에 더 집중할 수 있다.

- 또한 객체 간의 결합도를 낮춰서 코드 수정, 테스트, 유지보수가 쉬운 구조를 만들 수 있다.

## | IoC(Inversion of Control) - 제어의 역전

- IoC는 객체의 생성과 관리에 대한 제어권이 개발자에게서 Spring Container로 넘어가는 것을 의미한다.

- 일반적인 자바 코드에서는 개발자가 직접 new 키워드를 사용해서 객체를 생성하고 필요한 객체를 연결한다.

- Spring에서는 개발자가 직접 객체를 생성하고 관리하는 것이 아니라, Spring Container가 객체를 생성하고 필요한 곳에 주입한다.


### 왜 쓰는가?

- 객체 생성과 관리 책임을 개발자가 모두 담당하면 클래스 간의 의존성이 강해지고, 코드 변경이 어려워진다.

- IoC를 사용하면 객체 생성과 연결을 Spring이 담당하기 때문에 개발자는 비즈니스 로직에 더 집중할 수 있다.

<br>

---

## | DI(Dependency Injection) - 의존성 주입

- DI는 객체가 필요로 하는 의존 객체를 직접 생성하지 않고, 외부에서 주입받는 방식이다.

- 여기서 의존성이란 어떤 클래스가 동작하기 위해 필요한 다른 객체를 의미한다.

### 왜 쓰는가?

- 객체 내부에서 직접 new를 사용하면 특정 클래스에 강하게 의존하게 된다.

- DI를 사용하면 필요한 객체를 외부에서 넣어주기 때문에 클래스 간 결합도를 낮출 수 있다.

- DI는 코드를 더 유연하고 테스트하기 쉬운 구조로 만들기 위해 사용한다.

<br>

## | AOP(Aspect-Oriented Programming) - 관점 지향 프로그래밍

- AOP는 여러 클래스에서 반복적으로 사용되는 공통 기능을 분리해서 관리하는 프로그래밍 방식이다.

- 예를 들어 로그 출력, 인증 검사, 권한 확인, 트랜잭션 처리, 실행 시간 측정 같은 기능은 여러 메서드에서 반복될 수 있다.

- AOP는 이런 공통 관심사를 따로 분리해서 필요한 시점에 적용할 수 있도록 도와준다.

### 왜 쓰는가?

- 개발을 하다 보면 핵심 기능과는 직접 관련이 없지만 반드시 필요한 코드들이 반복적으로 등장한다.

- 공통 코드를 매번 직접 작성하면 중복이 많아지고 유지보수가 어려워진다.

- AOP를 사용하면 핵심 로직은 깔끔하게 유지하면서 공통 기능을 한 곳에서 관리할 수 있다.


<br>

## | POJO(Plain Old Java Object)

- 특별한 기술이나 프레임워크에 종속되지 않은 순수한 자바 객체를 의미한다.

- 특정 클래스를 상속받거나, 특정 인터페이스를 반드시 구현해야 하는 객체가 아니라 일반적인 자바 클래스를 말한다.

### 왜 쓰는가?

- 특정 프레임워크에 강하게 의존하는 코드는 나중에 프레임워크를 변경하거나 테스트할 때 어려움이 생긴다.

- POJO 방식으로 코드를 작성하면 순수 자바 코드에 가까운 구조를 유지할 수 있다.

- Spring은 이런 POJO 기반 개발을 지원하기 때문에 개발자가 복잡한 상속 구조 없이도 객체지향적인 코드를 작성할 수 있게 해준다.

<br>

## | Spring Bean

- Spring Bean은 Spring Container가 생성하고 관리하는 객체를 의미한다.

- 개발자가 직접 new로 만든 객체는 일반 객체이고, Spring Container에 등록되어 관리되는 객체는 Spring Bean이다.

### 왜 쓰는가?

- Spring이 객체를 관리하려면 먼저 그 객체를 Spring Container 안에 등록해야 한다.

- Bean으로 등록된 객체는 Spring이 생성, 의존성 주입, 생명주기 관리까지 담당할 수 있다.

- Bean으로 관리되기 때문에 필요한 곳에서 자동으로 주입받을 수 있고, 객체를 매번 새로 만들지 않아도 된다.

- 따라서 사람의 실수를 줄여주고 효율적으로 객체를 이용할 수 있다.

<br>

---

## | Bean Container

- Bean Container는 Spring Bean을 생성하고 관리하는 공간이다.

- Bean Container는 애플리케이션 실행 시 Bean 객체를 만들고, 필요한 의존성을 연결하고, 객체의 생명주기를 관리한다.

- 개발자가 직접 객체를 생성하고 연결하는 것이 아니라, Bean Container가 객체들을 관리하면서 필요한 곳에 주입해준다.

### 왜 쓰는가?

- 애플리케이션 규모가 커질수록 객체의 개수도 많아지고, 객체 간의 의존 관계도 복잡해진다.

- Bean Container를 사용하면 객체 생성과 의존성 연결을 Spring이 담당하기 때문에 코드의 구조가 단순해진다.

<br>

## | HTTP(HyperText Transfer Protocol)

- HTTP는 클라이언트와 서버가 데이터를 주고받기 위한 통신 규칙이다.

- 사용자가 웹 브라우저에서 페이지를 열거나, 로그인 버튼을 누르거나, 게시글 목록을 조회할 때 클라이언트와 서버는 HTTP를 기준으로 요청과 응답을 주고받는다.

- HTTP는 기본적으로 요청(Request)과 응답(Response) 구조로 동작한다.

### 왜 쓰는가?

- 클라이언트와 서버가 서로 아무 규칙 없이 데이터를 주고받으면, 어떤 요청인지 어떤 응답인지 해석하기 어렵다.

- 즉 클라이언트와 서버가 서로 이해하기 쉽고 빠르게 통신하기 위해 사용한다.

<br>

---

## | HTTP Message

- HTTP Message는 클라이언트와 서버가 주고받는 HTTP 데이터의 형식이다.

- Request Message는 클라이언트가 서버에게 보내는 요청 메시지이다.

- Response Message는 서버가 클라이언트에게 돌려주는 응답 메시지이다.

### 왜 쓰는가?

- HTTP Message를 사용하면 요청의 목적, 요청 대상, 데이터 형식, 인증 정보, 실제 전달할 데이터 등을 구조적으로 표현할 수 있다.

- 결국 HTTP Message는 클라이언트와 서버가 요청과 응답을 명확하게 해석하기 위한 데이터 전달 구조이다.

## | HTTP Method

- HTTP Method는 클라이언트가 서버에게 어떤 동작을 요청하는지 표현하는 방식이다.

- 대표적인 HTTP Method에는 GET, POST, PUT, PATCH, DELETE가 있다.

- HTTP는 기본적으로 요청(Request)과 응답(Response) 구조로 동작한다.

### 왜 쓰는가?

- 같은 URL을 사용하더라도 어떤 동작을 할 것인지 구분할 필요가 있다.

- HTTP Method를 사용하면 자원과 행위를 분리해서 표현할 수 있다.
