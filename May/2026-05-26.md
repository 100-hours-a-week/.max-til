## 🚀 Today I Learned

## | 데이터베이스

- 데이터베이스(Database)는 데이터를 체계적으로 저장하고 관리하기 위한 공간이다.

-  단순히 데이터를 저장하는 것이 아니라, 필요한 데이터를 빠르게 조회하고 수정하고 삭제할 수 있도록 구조화해서 관리한다.

- 회원 정보, 게시글, 댓글, 좋아요, 주문 내역처럼 서비스에서 계속 저장하고 활용해야 하는 데이터는 대부분 데이터베이스에 저장된다.

### 왜 쓰는가?

- 서비스가 커질수록 데이터를 단순 파일로 관리하면 중복, 누락, 검색 속도 문제, 데이터 불일치 문제가 생길 수 있다.

- 데이터베이스를 사용하면 데이터를 일정한 구조로 저장할 수 있고, 필요한 데이터를 조건에 맞게 빠르게 조회할 수 있다.

- 또한 여러 사용자가 동시에 데이터를 다루는 상황에서도 데이터가 꼬이지 않도록 관리할 수 있기 때문에 실제 서비스에서는 필수적으로 사용된다.

## | ERD

- ERD(Entity Relationship Diagram)는 데이터베이스의 테이블 구조와 테이블 사이의 관계를 그림으로 표현한 설계도이다.

- 어떤 테이블이 필요한지, 각 테이블에는 어떤 컬럼이 들어가는지, 테이블끼리는 어떤 관계를 가지는지 한눈에 볼 수 있게 해준다.

### 왜 쓰는가?

- 데이터베이스를 바로 만들기 시작하면 나중에 테이블 관계가 꼬이거나 중복 데이터가 생길 수 있다.

- ERD를 먼저 작성하면 데이터가 어떻게 연결되는지 미리 확인할 수 있고, 어떤 테이블에 외래 키를 둘지 고민할 수 있다.

- 또한 데이터를 삭제할 때 연관된 데이터도 함께 삭제해야 하는지, 아니면 삭제를 막아야 하는지 같은 정책도 미리 정리할 수 있다.

- 그래서 ERD는 단순한 그림이 아니라 백엔드 API, 데이터베이스 설계, 기능 구현의 기준이 되는 설계도라고 볼 수 있다.

## | Index

- Index는 데이터베이스에서 데이터를 더 빠르게 찾기 위해 사용하는 검색용 자료구조이다.

- 책에서 원하는 내용을 빠르게 찾기 위해 목차나 색인을 보는 것처럼, 데이터베이스도 특정 컬럼에 인덱스를 만들어두면 조건 검색을 더 빠르게 할 수 있다.

- 예를 들어 게시글 테이블에서 `created_at`, `user_id`, `title` 같은 컬럼을 자주 검색한다면 해당 컬럼에 인덱스를 걸 수 있다.

### 왜 쓰는가?

- 데이터가 적을 때는 전체 데이터를 하나씩 확인해도 큰 문제가 없지만, 데이터가 많아지면 전체 테이블을 처음부터 끝까지 읽는 방식은 매우 느려진다.

- 인덱스를 사용하면 데이터베이스가 모든 행을 검사하지 않고, 정렬된 구조나 검색 구조를 이용해 필요한 데이터가 있는 위치를 빠르게 찾을 수 있다.

- 하지만 인덱스는 추가 저장 공간을 사용하고, 데이터가 추가·수정·삭제될 때 인덱스도 함께 갱신되어야 한다.

- 그래서 인덱스는 무조건 많이 만드는 것이 아니라, 자주 조회되거나 정렬·검색 조건으로 자주 사용되는 컬럼에 신중하게 만들어야 한다.

## | Tree 와 Hash

- Tree와 Hash는 데이터를 빠르게 찾기 위해 사용하는 대표적인 자료구조이다.

- Tree는 데이터를 정렬된 구조로 저장하여 범위 검색이나 정렬된 조회에 유리하다.

- Hash는 특정 값을 해시 함수로 변환해서 저장 위치를 찾는 방식으로, 정확히 일치하는 값을 빠르게 찾는 데 유리하다.

### 왜 쓰는가?

- 데이터베이스는 단순히 데이터를 저장하는 것에서 끝나는 것이 아니라, 필요한 데이터를 빠르게 찾을 수 있어야 하기 때문이다.

- Tree는 범위를 기준으로 찾을 때 유리하고 Hash는 정확히 같은 값을 찾는 상황에서 빠르게 동작할 수 있다.



<br>

## | B-Tree

- B-Tree는 데이터베이스 인덱스에서 자주 사용되는 트리 기반 자료구조이다.

- 데이터를 정렬된 상태로 유지하면서 검색, 삽입, 삭제를 효율적으로 처리할 수 있다.

- 하나의 노드가 여러 개의 키를 가질 수 있기 때문에 트리의 높이를 낮게 유지할 수 있다.

### 왜 쓰는가?

- 데이터베이스는 디스크에서 데이터를 읽는 비용이 크기 때문에, 최대한 적은 횟수로 원하는 데이터에 접근해야 한다.

- B-Tree는 트리의 높이를 낮게 유지해서 검색할 때 많은 단계를 거치지 않도록 도와준다.



<br>

## | B+Tree

- B+Tree는 B-Tree를 데이터베이스 인덱스에 더 적합하게 변형한 자료구조이다.

- 실제 데이터나 데이터의 위치 정보는 리프 노드에 모아두고, 내부 노드는 길을 안내하는 키 역할을 한다.

- 리프 노드끼리는 연결되어 있어 범위 검색을 할 때 순차적으로 데이터를 읽기 좋다.


### 왜 쓰는가?

- 데이터베이스에서는 단일 값 검색도 중요하지만, 특정 범위의 데이터를 빠르게 조회하는 경우도 많다.

- 예를 들어 `created_at` 기준으로 최근 게시글을 조회하거나, 가격이 특정 범위 안에 있는 상품을 찾는 경우가 있다.

- B+Tree는 리프 노드에 데이터가 정렬되어 있고 서로 연결되어 있기 때문에 범위 검색과 정렬 조회에 유리하다.


<br>

## | Hash

- Hash는 값을 해시 함수에 넣어 특정 위치를 계산하고, 그 위치를 기준으로 데이터를 찾는 방식이다.

- 예를 들어 `email = test@test.com`처럼 정확히 같은 값을 찾을 때 빠르게 접근할 수 있다.

- 하지만 데이터가 정렬된 구조가 아니기 때문에 범위 검색에는 적합하지 않다.


### 왜 쓰는가?

- 특정 값을 정확히 찾는 조회에서는 Tree보다 빠르게 동작할 수 있기 때문이다.

- 예를 들어 로그인할 때 이메일이 정확히 일치하는 사용자를 찾는 상황에서는 Hash 방식이 유리할 수 있다.

## 💭 오늘의 회고

오늘은 데이터베이스에서 자주 등장하는 기본 개념들을 정리해보았다. 처음에는 데이터베이스를 단순히 데이터를 저장하는 공간 정도로만 생각했지만, 실제로는 데이터를 어떻게 구조화하고, 어떻게 빠르게 조회하고, 어떻게 안전하게 변경할 것인지까지 함께 고민해야 한다는 것을 알게 되었다.

특히 ERD를 공부하면서 테이블을 만드는 것보다 먼저 데이터 사이의 관계를 설계하는 것이 중요하다고 느꼈다. 프로젝트에서 게시글, 댓글, 좋아요, 사용자 같은 기능을 만들 때 각각의 테이블이 따로 존재하는 것이 아니라 서로 FK로 연결되고, 삭제나 수정 상황에서도 영향을 주고받기 때문에 처음 설계가 정말 중요하다는 생각이 들었다.

Index도 단순히 “검색을 빠르게 해주는 것”이라고만 알고 있었는데, 인덱스가 Tree나 Hash 같은 자료구조와 연결되어 있다는 점이 흥미로웠다. 특히 어떤 검색은 Tree가 유리하고, 어떤 검색은 Hash가 유리하다는 것을 보면서 인덱스도 무조건 추가하는 것이 아니라 조회 패턴에 맞게 설계해야 한다는 것을 알게 되었다.

마지막으로 Transaction을 공부하면서 데이터베이스에서 안정성이 왜 중요한지 다시 생각하게 되었다. 단순히 SQL을 실행하는 것이 아니라, 여러 작업이 모두 성공해야만 반영되고 문제가 생기면 되돌리는 구조가 있어야 실제 서비스에서 데이터가 꼬이지 않을 수 있다. 앞으로 프로젝트를 진행할 때 단순히 기능이 동작하는지만 보는 것이 아니라, 데이터가 안전하게 저장되고 변경되는지도 함께 고민해야겠다.

앞으로 트랜잭션을 활용할 곳이 어디에있을까 다시한번 고민하며 이번주 과제에서도 활용을 해볼 생각이다.
