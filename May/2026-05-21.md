## 🚀 Today I Learned

## | Stateful

- Stateful은 서버가 클라이언트의 이전 상태를 기억하면서 통신하는 방식이다.

- 클라이언트가 한 번 요청한 정보나 로그인 상태, 진행 중인 작업 정보를 서버가 저장하고 있다가 다음 요청에서 이어서 사용할 수 있다.

- 예를 들어 세션 기반 로그인은 서버가 사용자의 로그인 상태를 기억하기 때문에 Stateful 방식에 가깝다.

### 왜 쓰는가?

- 사용자의 이전 요청 정보를 계속 활용해야 하는 서비스에서는 상태를 저장하는 방식이 필요하기 때문이다.

- 예를 들어 로그인 후 장바구니를 사용하거나, 사용자가 어디까지 작업했는지 이어서 처리해야 하는 경우 서버가 상태를 기억하면 구현이 편리하다.

- 하지만 서버가 사용자의 상태를 관리해야 하기 때문에 사용자가 많아질수록 서버 부담이 커질 수 있다.

## | Stateless

- Stateless는 서버가 클라이언트의 이전 상태를 기억하지 않는 통신 방식이다.

- 클라이언트가 요청할 때마다 필요한 정보를 함께 보내야 하며, 서버는 각 요청을 독립적으로 처리한다.

- HTTP는 기본적으로 Stateless한 특징을 가진다.

### 왜 쓰는가?

- 서버가 클라이언트의 상태를 기억하지 않으면 요청을 처리하는 구조가 단순해진다.

- 또한 어떤 서버가 요청을 받아도 필요한 정보가 요청 안에 들어있기 때문에 여러 서버로 트래픽을 분산하기 쉽다.

- 그래서 많은 사용자가 접속하는 웹 서비스에서는 Stateless 구조가 확장성 측면에서 유리하다.

## | 로드 밸런서

- 로드 밸런서는 여러 서버 앞에서 사용자의 요청을 적절한 서버로 나누어 보내는 장치 또는 기술이다.

- 하나의 서버에 모든 요청이 몰리면 서버가 느려지거나 장애가 발생할 수 있다.

- 로드 밸런서는 요청을 여러 서버에 분산시켜 전체 서비스가 안정적으로 동작하도록 도와준다.

### 왜 쓰는가?

- 사용자가 많아지면 하나의 서버만으로 모든 요청을 처리하기 어렵다.

- 이때 여러 대의 서버를 두고 로드 밸런서가 요청을 나누어 보내면 서버의 부담을 줄일 수 있다.

- 또한 특정 서버에 장애가 발생했을 때 정상 서버로 요청을 보내 서비스 중단을 줄일 수 있다.

## | Connectless

- Connectless는 데이터를 보내기 전에 별도의 연결을 맺지 않고 바로 데이터를 전송하는 방식이다.

- 대표적으로 UDP가 Connectless 방식에 해당한다.

- 연결 확인 과정이 없기 때문에 속도는 빠르지만, 데이터가 반드시 도착한다는 보장은 상대적으로 약하다.

### 왜 쓰는가?

- 모든 통신에서 안정성보다 속도가 더 중요한 경우가 있다.

- 예를 들어 실시간 영상 스트리밍이나 게임처럼 약간의 데이터 손실보다 빠른 전송이 중요한 상황에서는 연결 과정이 오히려 부담이 될 수 있다.

- 그래서 Connectless 방식은 빠른 전송이 중요한 통신에서 사용된다.

## | 인증

- 인증(Authentication)은 사용자가 누구인지 확인하는 과정이다.

- 예를 들어 로그인할 때 아이디와 비밀번호를 입력하고, 서버가 이 정보가 맞는지 확인하는 과정이 인증이다.

- 인증이 성공하면 서버는 이 사용자가 실제 등록된 사용자인지 판단할 수 있다.

### 왜 쓰는가?

- 서비스는 요청을 보낸 사용자가 누구인지 알아야 개인화된 기능을 제공할 수 있다.


<br>

## | 인가

- 인가(Authorization)는 인증된 사용자가 특정 기능이나 자원에 접근할 권한이 있는지 확인하는 과정이다.

- 인증이 “누구인지 확인”하는 것이라면, 인가는 “무엇을 할 수 있는지 확인”하는 것이다.

- 예를 들어 로그인한 사용자라도 다른 사람의 게시글을 삭제할 수 없다면, 이 부분은 인가에서 막아야 한다.

### 왜 쓰는가?

- 모든 사용자가 모든 기능을 사용할 수 있으면 보안 문제가 발생할 수 있다.

- 사용자마다 접근 가능한 데이터와 기능이 다르기 때문에 권한을 확인하는 과정이 필요하다.

- 그래서 인가는 인증된 사용자가 허용된 작업만 수행하도록 제한하기 위해 사용된다.


<br>

## | 쿠키

- 쿠키(Cookie)는 서버가 클라이언트의 브라우저에 저장해두는 작은 데이터이다.

- 서버는 응답을 보낼 때 쿠키를 함께 보낼 수 있고, 브라우저는 이후 요청마다 해당 쿠키를 자동으로 함께 보낼 수 있다.

- 로그인 상태 유지, 사용자 설정 저장, 장바구니 정보 저장 등에 사용될 수 있다.

### 왜 쓰는가?

- HTTP는 기본적으로 Stateless이기 때문에 서버는 이전 요청을 기억하지 않는다.

- 하지만 로그인 상태처럼 요청 간에 유지해야 하는 정보가 필요할 때가 있다.

- 쿠키를 사용하면 브라우저가 요청마다 특정 정보를 함께 보내기 때문에, 서버가 사용자를 구분하거나 상태를 이어서 처리할 수 있다.


<br>

## | 세션

- 세션(Session)은 사용자의 상태 정보를 서버 쪽에 저장하는 방식이다.

- 사용자가 로그인하면 서버는 세션 저장소에 로그인 정보를 저장하고, 클라이언트에게는 세션을 식별할 수 있는 값만 전달한다.

- 이후 클라이언트가 요청할 때 세션 ID를 보내면, 서버는 이 값을 기준으로 사용자의 상태를 확인한다.

### 왜 쓰는가?

- 로그인 정보나 사용자 상태를 클라이언트에 직접 많이 저장하면 보안상 위험할 수 있다.

- 세션을 사용하면 중요한 상태 정보는 서버에 저장하고, 클라이언트는 식별자만 가지고 있게 할 수 있다.

- 그래서 서버가 사용자의 로그인 상태를 안전하게 관리해야 할 때 세션을 사용한다.

## | 페이징

- 페이징(Paging)은 많은 데이터를 한 번에 모두 가져오지 않고, 일정한 단위로 나누어 조회하는 방식이다.

- 예를 들어 게시글이 10,000개 있어도 처음에는 10개만 가져오고, 사용자가 더 요청하면 다음 데이터를 가져오는 방식이다.

- 게시글 목록, 댓글 목록, 상품 목록처럼 데이터가 많은 화면에서 자주 사용된다.

### 왜 쓰는가?

- 많은 데이터를 한 번에 조회하면 서버, 데이터베이스, 클라이언트 모두 부담이 커진다.

- 페이징을 사용하면 필요한 만큼만 데이터를 가져올 수 있어 응답 속도를 높이고 자원을 절약할 수 있다.

- 또한 사용자는 한 번에 너무 많은 데이터를 보는 것보다 필요한 만큼 나누어 보는 것이 더 편하다.


<br>

## | 페이징 구현 방식

- 페이징 구현 방식에는 대표적으로 Offset 기반, Page 기반, Key 기반, Cursor 기반, Token 기반 방식이 있다.

- Offset 기반은 몇 번째 데이터부터 가져올지 기준을 정하는 방식이고, Page 기반은 페이지 번호를 기준으로 데이터를 조회하는 방식이다.

- Key 기반은 마지막으로 조회한 데이터의 기준값을 넘겨 다음 데이터를 조회하는 방식이며, Cursor 기반은 이 기준값을 커서라는 형태로 만들어 다음 요청에 사용하는 방식이다.

- Token 기반은 커서나 조회 조건을 토큰처럼 감싸서 클라이언트에게 전달하는 방식이다.
