## 🚀 Today I Learned

## | 파일처리

- 파일처리는 사용자가 업로드한 이미지,문서,첨부파일 같은 데이터를 서버가 받아서 저장하고, 나중에 다시 사용할 수 있도록 관리하는 과정이다.

- 이미지는 단순한 문자열 데이터가 아니기 때문에 보통 multipart/form-data 형식으로 전송하여 multipartFile로 받는다

### 왜 쓰는가?

- 이미지 파일 자체로 DB에 직접 저장하는건 효율적이지 않은 방법이다.

- 파일은 크기가 상대적으로 크고 DB가관리하기에 부담이 큰 작업이기 때문이다.

- 즉 DB에는 파일자체를 넣는것이 아닌 파일은 따로(S3,local storage) 저장하고 DB에는 접근할 수 있는 URL만 저장한다.

## | 예외처리

- 예외 처리는 애플리케이션 실행 중 문제가 발생했을 때 서버가 무작정 죽거나 이상한 응답을 보내지 않도록 정해진 형식으로 에러를 처리하는 구조

- 각 컨트롤러나 서비스에서 매번 try-catch로 반복하면 코드가 복잡해지기 때문에 GlobalExceptionHandler를 만들어 예외를 한곳에서 처리하는게 좋다.

### 왜 쓰는가?

- 예외처리를 공통화시키면 컨트롤러와 서비스는 비즈니스 로직에만 집중할 수 있기 때문이다.

- 또한 ErrorCode같은 응답을 정하여 클라이언트에게 일관된 에러 응답을 반환하면서 에러 상황도 알아보기가 쉬워진다.

## | 필터(Filter)

- 필터는 클라이언트의 요청이 Spring MVC의 컨트롤러에 도달하기 전에 먼저 실행되는 영역

- DispatcherServlet보다 앞단에서 실행된다.

- 즉, 사용자의 요청이 들어오면 컨트롤러보다 먼저 필터가 요청을 가로채서 검사할 수 있다

### 왜 쓰는가?
- 모든 요청에 대해 공통적으로 처리해야하는 작업에 적합하기 때문에

- JWT 인증 처리나, 인코딩 처리, 보안 관련 검사, 파일 압축하는데 쓰임

- Spring 서버의 부하를 줄이고 인증이 안된 쓸데없는 요청을 저 앞단에서 미리 처리할 수 있기 때문

<br>

## | 인터셉터(Interceptor)

- 인터셉터는 Spring MVC 내부에서 컨트롤러가 실행되기 전후에 요청을 가로채는 기능

- 필터와 다르게 Spring 내부에서 동작한다.

### 왜 쓰는가?

- Spring 내부에서 동작하기 때문에 컨트롤러와 관련된 공통 작업을 처리하기에 용이하기 때문이다.

## 💭 오늘의 회고

오늘은 파일처리,예외처리,필터,인터셉터에 대해 배웠다. 오늘 배운 내용들은 거의 내가 개발할 때 고민을 많이 한 내용들이었는데 결국 개발은 기능을 한번에 몰아서 구현하는 것이 아닌 각 기능의 책임을 분리하는 방식으로 설계해야 좋겠구나를 다시 한번 느꼈다. 
예를 들어 게시글 등록이나 프로필 이미지 등록을 하나의 API가 담당하게 만들었었는데 이렇게 하다보니 나중에 이미지 미리보기나, 이미지가 여러 장이 될 수 있도록 추가할때 확장성이 안좋다는 것을 실감하였다.
필터와 인터셉터는 둘 다 요청을 가로채는 역할을 하지만 실행 위치와 사용목적이 다르다는 점이 중요했다. 개발을 할 때 필터의 개념을 모르고 인터셉터를 사용했는데 이번 강의를 들으며 내가 어떤 로직을 어디에 사용해야 할지 다시한번 고민하게 만들었고 오늘 이 부분에 대해서 정리를 하고 코드를 수정해볼 계획이다.
항상 느끼지만 동작하는 코드가 중요한게 아닌 확장성과 유지보수를 고려하고 각 로직이 어떤 책임을 가지고 어디서 어떻게 동작하는지 흐름을 알고 개발을 해야겠다고 생각하였다.
