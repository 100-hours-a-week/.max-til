## | JPA의 3가지 값 타입

1. 기본값 타입: 자바의 기본 타입(`int`, `double`), 래퍼 클래스(`Integer`, `Long`), `String` 등

2. 임베디드 타입(Embedded Type, 복합 값 타입): 새로운 클래스를 직접 정의해서 사용하는 값 타입 (예: `Address(city, street, zipcode)`, `Period(startDate, endDate)`)

3. 컬렉션 값 타입 : 값 타입을 하나가 아니라 컬렉션(`List`, `Set`)에 담아 다루는 형태

### 왜 쓰는가?

- 기본값 타입 : 식별자나 연관관계가 필요 없는 단순한 값을 엔티티 내부에서 직접 다루어 설계를 단순화하고, 데이터베이스와 매핑하기 위해서

- 임베디드 타입 : 작성자 정보, 생성 일시처럼 의미 있는 값들의 묶음을 각각 하나의 객체로 만들어 반복을 줄이고 설계를 일관되게 유지하기 위해서

- 컬렉션 값 타입 : 여러 개의 단순 값을 엔티티 없이 테이블에 분리 저장하면서도 부모 객체와 함께 일괄적으로 관리하기 위해서

## | JPQL

- 데이터베이스 테이블이 아닌 엔티티 객체를 대상으로 쿼리하는 객체지향 쿼리 언어이다.

### 왜 쓰는가?
- JPA의 기본 조회 기능(`find()`)만으로는 복잡한 검색 조건이나 통계, JOIN 쿼리를 처리하기 어렵기 때문이다.

- SQL 문법과 매우 유사하지만, 특정 DB 방언(Dialect)에 의존하지 않고 객체 중심의 추상화된 쿼리를 작성할 수 있다는 장점이 있다.

- 그러나 결국 문자열 기반이기 때문에 컴파일 시점에 문법 에러를 잡지 못한다는 치명적인 단점이 있다

## | Spring Data JPA

- JPA 기반의 데이터 처리를 자동화해 인터페이스만으로 리포지토리를 쉽게 구현할 수 있도록 도와주는 Spring 모듈

- JpaRepository 인터페이스를 상속하면 JPA 표준 기능을 그대로 사용하면서도 CRUD 메서드, 페이징, 정렬 같은 공통 데이터 처리 로직이 기본 제공되며, 메서드 이름만으로 조건이 포함된 조회를 수행하는 쿼리 메서드 기능까지 활용할 수 있다.

1. 쿼리 메소드
    - 인터페이스에 규칙에 맞는 메서드 이름만 정의하면 스프링이 내부적으로 쿼리를 자동 생성해줌

    - 단순 조건 검색 시, 복잡한 JPQL을 작성하지 않고 단 한 줄의 메서드 선언만으로 처리가 가능해 생산성이 높아짐

2. @Query
    - 메서드 이름으로 풀기 어려운 복잡한 쿼리나 JOIN이 필요할 때, 메서드 상단에 `@Query` 어노테이션을 붙여 JPQL을 직접 명시하는 기능이다.

    - 조건이 3개 이상 늘어나 메서드 이름이 너무 길어질 때 가독성을 챙길 수 있고, 성능 최적화를 위한 쿼리를 세밀하게 짤 수 있다.

3. @EntityGraph
    - 연관된 엔티티들을 조회할 때, JPA의 FetchType 전략(지연 로딩)을 무시하고 한 번에 함께 조회(Fetch Join)하도록 돕는 어노테이션

    - 엔티티 탐색 시 발생하는 대표적인 성능 문제인 N+1 문제를 깔끔하게 해결하기 위해서 사용됨, 그러나 현실적으로 잘 쓰지 않는 방법

4. Pageable
    - 파라미터에 `Pageable`을 넘기고 반환 타입으로 `Page`나 `Slice`를 받으면 스프링이 페이징 처리를 자동으로 해주는 기능이다.

5. 사용자 정의 레포지토리
    - Spring Data JPA 인터페이스 방식(`JpaRepository`)만으로 해결되지 않는 복잡한 쿼리나 동적 쿼리를 직접 구현하기 위해, 별도의 사용자 정의 인터페이스와 구현 클래스를 조합하는 방식
### 왜 쓰는가?

- 반복적인 CRUD 코드 없이 인터페이스만으로 리포지토리를 구현해 개발 생산성과 유지보수 효율을 높이기 위해서

- 개발 속도를 크게 높일 뿐만 아니라, 리포지토리 구조를 표준화해 팀원 간 구현 방식 차이로 인한 유지보수 부담을 줄일 수 있음

## | QueryDSL

- QueryDSL은 JPQL의 문자열 형태 코딩 방식에서 벗어나, 자바 코드로 안전하게 쿼리를 작성할 수 있도록 돕는 오픈소스 빌더 프레임워크이다. 

- (컴파일 타임에 엔티티 구조를 기반으로 Q-Class를 생성하여 사용)

### 왜 쓰는가?

- 동적 쿼리 작성이 압도적으로 편리해진다.

- 문법 오류를 컴파일 시점에 잡아내기 위해서이다.

- 자동 완성 기능의 도움을 받아 복잡한 통계나 다중 JOIN 쿼리를 안전하고 직관적으로 짤 수 있다.

## 💭 오늘의 회고

오늘은 JPA를 사용할 때 실제 프로젝트에서 데이터를 어떻게 더 객체지향적으로 다루고, 복잡한 조회들을 어떻게 뽑아낼 수 있을지에 대해 배웠다.

JPQL을 정리하면서 JPA가 단순히 findById()만 제공하는 것이 아니라 엔티티 객체를 대상으로 SQL과 비슷한 쿼리를 작성할 수 있게 해준다는 점을 이해했다.

하지만 문자열 기반이기 때문에 오타나 문법 오류를 컴파일 시점에 잡기 어렵워서 실무에서는 단순 쿼리는 Spring Data JPA를 많이 이용하고 

복잡하거나 도적인 조건이 많아질수록 QueryDSL 같은 라이브러리를 이용한다는 것을 알았다.

앞으로 과제를 수행할 때 어떤 상황일 때 어떠한 방식이 더 잘 어울리는지 고민하고 배운대로 적용을 해야겠다는 생각을 하였다.
